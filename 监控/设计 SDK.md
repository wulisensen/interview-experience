- 如果让你主导开发性能指标或错误监控 SDK，会考虑哪些方面？
  - 1、监控指标体系
    - 性能监控
      - 核心 Web 指标：FCP、CLS、FID
      - 其他：FCP、接口耗时、资源耗时
    - 错误监控
      - 运行时错误：JS Error/Promise Rejection
      - 资源加载错误：图片、样式、脚本
      - 接口错误：HTTP 状态码异常或业务逻辑错误
    - 用户行为
      - PV\UV：基础访问
      - 行为轨迹：用户在报错前做了什么（点了那个按钮、跳转了哪个页面）
  - 2、核心架构
    需要可扩展性：要适配 web、nodejs、小程序
    - 插件化机制：将核心逻辑（调度、上报）与采集逻辑（性能指标、错误捕获）解耦
      - 好处：开发者可以根据需要按需引入插件，减小包体积
    - 生命周期勾子：提供 onBeforeSend、onAfterSend 等钩子，允许用户在数据发送前进行脱敏处理或二次加工
  - 3、高性能上报粗略
    - 异步非阻塞：
      - 优先 sendBeacon ，它能在页面关闭时异步发送数据，且不占用主线程资源。
      - 降级方案：Image 或者 fetch/XHR
    - 批量与限流
      - 聚合发送：设置缓冲区，定时或者达到最大数量合并发送，减少 http 请求
      - 优先级管理：错误日志立即发送，性能日志延迟发送
    - 抽样采集
      - 大流量应用，通过配置抽样率来降低服务的压力
  - 4、辅助排查
    - Sourcemap
    - 自动采集设备信息（OS、浏览器版本）、网络状态、URL 路径
    - 录屏还原：在某些错误发生时，记录操作的 dom 变化快照，像看视频一样复习bug
  - 5、安全与稳定性
    - 错误自愈：SDK 内部必须包裹 try-catch，确保 SDK 自身的崩溃不会影响主业务流程
    - 隐私脱敏：自动处理敏感字段（密码、手机号），确保符合合规要求
    - 离线存储：如果网络中断，可以将日志临时存入 indexDB 或者 localStorage，待网络恢复后续传
